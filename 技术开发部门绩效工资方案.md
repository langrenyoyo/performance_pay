# 技术开发部门绩效工资方案

版本：V1.0  
适用部门：技术开发部  
适用对象：开发工程师、测试工程师、运维/DevOps、技术负责人及相关技术岗位  
生效日期：____年__月__日

## 一、方案目的

为建立公平、透明、可执行的技术开发部门绩效工资机制，鼓励员工高质量完成项目交付，提升系统稳定性、研发效率和团队协作水平，特制定本方案。

本方案坚持以下原则：

1. 稳定收入为主，绩效激励为辅。
2. 以结果和质量为核心，不以代码行数、加班时长、在线时长作为单一考核依据。
3. 考核过程应有目标、有记录、有反馈、可申诉。
4. 对需求变更、外部依赖、公司战略调整等非员工个人原因导致的影响，应合理区分责任。

## 二、适用范围

本方案适用于公司技术开发部门以下岗位：

1. 前端开发工程师
2. 后端开发工程师
3. 移动端开发工程师
4. 测试工程师
5. 运维/DevOps 工程师
6. 数据库/中间件/平台工程师
7. 技术负责人、项目技术负责人及其他技术岗位

特殊岗位可在本方案基础上另行制定补充考核细则。

## 三、薪资结构

技术开发岗位月度薪资建议由固定工资、绩效工资及其他奖金组成。

| 薪资项 | 建议占比 | 说明 |
| --- | ---: | --- |
| 固定工资 | 80%-90% | 按劳动合同约定正常发放，保障员工基本收入稳定 |
| 月度绩效工资 | 按岗位或员工约定金额 | 根据个人绩效结果浮动发放 |
| 项目奖金/季度奖金 | 视公司制度确定 | 用于奖励重大项目、关键版本、技术突破等 |
| 年终奖 | 视公司经营情况确定 | 与公司经营结果、部门贡献、个人年度绩效挂钩 |

原则上，技术开发岗位应以固定工资保障稳定收入，绩效工资金额应在薪资确认文件、劳动合同附件或公司薪酬制度中明确。

## 四、绩效工资计算方式

月度绩效工资按照以下公式计算：

```text
月度实发绩效工资 = 绩效工资 × 绩效总分 / 100
```

其中：

```text
绩效工资 = 公司与员工确认的当月绩效工资金额
绩效总分 = 各考核维度评分 × 对应权重后的加权总分
```

示例：

```text
绩效工资：3,000 元
绩效总分：86 分
月度实发绩效工资：3,000 × 86 / 100 = 2,580 元
```

## 五、个人绩效等级

个人绩效等级按照员工当期项目量与工作情况、代码质量、响应时效及其他工作完成情况综合评定。

| 等级 | 占比建议 | 说明 |
| --- | ---: | --- |
| S 卓越 | 5%-10% | 明显超出目标，解决关键问题，产生团队级或公司级影响 |
| A 优秀 | 20%-30% | 高质量完成目标，并有主动贡献 |
| B 合格 | 50%-60% | 正常完成岗位职责和项目交付 |
| C 待改进 | 5%-15% | 交付延期、质量不足或协作问题较明显 |
| D 不合格 | 原则上少量 | 严重失职、重大事故、长期未达岗位要求 |

B 级为正常合格表现，不应视为负面评价。C 级及以下绩效结果应有明确事实依据和沟通记录。绩效等级用于反馈、复盘、调薪、晋升和奖金参考，不作为当月绩效工资金额的乘数。

## 六、考核维度与权重

技术开发岗位绩效考核采用综合评分制，满分 100 分。各岗位统一采用以下权重：

| 考核维度 | 权重 | 考核重点 |
| --- | ---: | --- |
| 项目量与工作情况 | 70% | 项目任务量、工作饱和度、任务完成度、推进难度和整体工作表现 |
| 代码质量 | 15% | 代码可维护性、代码评审质量、缺陷率、规范性和技术债控制 |
| 响应时效 | 10% | 问题响应速度、沟通反馈及时性、线上支持和协作响应 |
| 其他工作完成情况 | 5% | 文档沉淀、临时任务、知识分享、流程配合和其他安排事项 |

单项绩效工资按照以下方式折算：

```text
单项绩效工资 = 绩效工资 × 该项权重 × 该项评分 / 100
```

示例：项目量与工作情况权重为 70%，若绩效工资为 3,000 元，该项评分为 80 分，则该项绩效工资为：

```text
3,000 × 70% × 80 / 100 = 1,680 元
```

不建议将以下指标作为单一或核心考核依据：

1. 代码行数
2. Git 提交次数
3. 加班时长
4. 在线时长
5. 单纯需求数量

上述指标可作为辅助参考，但不得替代对业务价值、交付质量和技术贡献的综合判断。

## 七、岗位适用说明

本方案对技术开发部门各岗位采用统一评分权重。不同岗位的工作内容差异，应体现在“项目量与工作情况”和“其他工作完成情况”的事实依据中。

示例：

1. 开发工程师重点记录需求数量、复杂度、交付质量和联调配合情况。
2. 测试工程师重点记录测试任务量、缺陷发现质量、回归覆盖和发布保障情况。
3. 运维/DevOps 工程师重点记录发布支持、系统保障、故障响应和自动化建设情况。
4. 技术负责人重点记录项目推进、技术方案、团队协作和风险处理情况。

## 八、加分项与扣分项

### 8.1 加分项

| 行为 | 加分建议 |
| --- | ---: |
| 解决核心技术难题 | +5 到 +15 分 |
| 主动发现并避免重大风险 | +5 到 +10 分 |
| 优化性能、成本、稳定性且效果明显 | +5 到 +15 分 |
| 帮助新人、输出高质量技术文档 | +3 到 +8 分 |
| 推动自动化、工程效率提升 | +5 到 +10 分 |

### 8.2 扣分项

| 行为 | 扣分建议 |
| --- | ---: |
| 无合理原因导致关键任务延期 | -5 到 -15 分 |
| 代码评审多次出现低级问题 | -3 到 -10 分 |
| 线上事故责任明确 | -10 到 -30 分 |
| 隐瞒风险、问题不上报 | -10 到 -20 分 |
| 严重协作问题并影响项目推进 | -5 到 -20 分 |

加分和扣分均应基于事实记录，不得仅凭主观印象处理。

## 九、绩效考核周期

绩效考核分为月度、季度和年度三个层次。

| 周期 | 用途 | 说明 |
| --- | --- | --- |
| 月度绩效 | 绩效工资发放 | 重点考察当月目标完成、交付质量和协作表现 |
| 季度复盘 | 晋升、调薪、奖金参考 | 重点考察阶段性成果、能力成长和项目贡献 |
| 年度评估 | 年终奖、岗位级别、长期激励参考 | 重点考察年度贡献、稳定表现和长期价值 |

## 十、绩效流程

1. 月初目标确认：直属负责人和员工确认本月工作目标、重点任务和交付标准。
2. 月中过程同步：对需求变化、资源不足、外部依赖、风险事项进行同步和记录。
3. 月末员工自评：员工根据当月目标完成情况提交自评。
4. 直属负责人初评：负责人结合事实记录、项目数据和协作反馈进行评分。
5. 部门绩效校准：部门负责人对不同团队、不同负责人评分尺度进行统一校准。
6. 结果反馈沟通：负责人向员工说明绩效等级、评分依据和改进建议。
7. 申诉与复核：员工对绩效结果有异议的，可在规定期限内申请复核。
8. 绩效归档：人事或部门负责人保存目标、评分、反馈和沟通记录。

## 十一、申诉机制

员工对绩效结果有异议的，可在绩效结果告知之日起 3 个工作日内提出书面申诉。

申诉处理流程：

1. 员工提交申诉理由及相关证明材料。
2. 直属负责人进行初步说明。
3. 部门负责人和人事部门共同复核。
4. 必要时组织绩效复议会议。
5. 在 5 个工作日内向员工反馈复核结果。

复核结果应以事实记录、项目资料、沟通记录和考核标准为依据。

## 十二、特殊情况处理

以下情况应在绩效评估中单独说明，不宜简单归责员工个人：

1. 产品需求在开发过程中发生重大变更。
2. 客户或外部系统接口延期。
3. 公司战略调整导致项目暂停、变更或取消。
4. 跨部门资源不足导致任务受阻。
5. 技术方案经集体评审通过后，因客观限制未达预期。

员工因请假、入职、离职、岗位调整等导致当月实际工作周期不完整的，绩效工资可按公司薪酬制度和实际出勤情况折算。

## 十三、合规与制度管理

本方案涉及员工劳动报酬和绩效考核，应按照公司制度管理流程执行。

公司应将本方案通过员工手册、制度公告、邮件、内部系统或签字确认等方式向员工公示或告知。涉及劳动合同约定薪资结构调整的，应依法与员工协商确认。

绩效工资发放和调整应符合国家及地方劳动用工、工资支付相关规定，不得以绩效考核名义随意克扣员工依法应得工资。
